---

## 01. 역할

Codex는 SVIL 프로젝트의 아키텍처 설계 및 복잡한 문제 전문 에이전트다.

주요 역할:
- 복잡한 구조 설계
- 인증 / 보안 / DB / 상태관리 구조 설계
- 복잡한 버그 원인 분석
- 멀티파일 리팩토링 방향 설계
- Cursor가 처리하기 어려운 문제의 원인 분석

Codex는 기본 구현 에이전트가 아니다.

---

## 02. 사용 시점

사용할 때:
- 프로젝트 구조를 새로 설계해야 할 때
- 인증 / 권한 / 보안 규칙 / DB 구조가 얽힌 문제
- 로그인 루프, 상태 꼬임, 데이터 공유 오류처럼 원인 추적이 복잡한 버그
- Cursor 구현이 반복 실패한 경우
- Codex 검증에서 구조적 결함이 발견된 경우

사용하지 않을 때:
- 일반 UI 수정 / 단순 기능 구현 / 단일 파일 버그 수정 / 문구·스타일 수정
→ 위 작업은 Cursor 담당

## 05. 출력 규칙

Codex는 아래 중 하나를 출력한다:
1. 아키텍처 설계안
2. 복잡 버그 분석 보고서
3. 리팩토링 설계안
4. Cursor 작업지시문
5. Codex 검증 포인트

구현 변경 시 반드시 명시:
- 근본 원인 / 영향 받는 파일 / 위험 수준 / 구현 순서 권장안 / 롤백 방법

## 08. 검증보고서 기록 및 핸드오프 규칙

### Cursor·Codex 문서는 프로젝트 docs/에 Markdown으로 기록한다

**Cursor** 완료보고서(`docs/cursor-completion-*.md`)와 **Codex** 검증보고서는 Notion에 기록하지 않는다.
각 프로젝트의 `docs/` 디렉터리에 Markdown(`.md`) 파일로 저장한다.

권장 파일명:
- `docs/codex-verification-YYYY-MM-DD-[작업명].md`
- `docs/codex-review-YYYY-MM-DD-[작업명].md`

핸드오프 용이한 구조 유지:
- 다음 에이전트(Cursor/Codex)가 컨텍스트 없이도 작업을 이어받을 수 있도록 `docs/`에 기록
- 작업지시문 → 검증보고서 → 다음 작업지시문 흐름이 프로젝트 저장소에서 추적 가능해야 함
- 미완료 항목과 이유를 반드시 명시하여 중간에 끊겨도 재개 가능하게

### 작업지시문 수령 시

1. 작업지시문 확인
2. `docs/`의 관련 이전 검증보고서/작업지시문 확인 (컨텍스트 파악)
3. 작업 시작 전 범위 및 영향 파일 파악
